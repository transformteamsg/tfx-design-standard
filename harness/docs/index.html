TFX Design Harness

A Claude Code plugin that wraps an AI agent so that everything it designs for the Teacher & School portfolio complies with the TFX Design Standard — not by asking nicely, but by structure: a control catalog it must satisfy, a loop with human gates, and checks that always run.

Introduction #

The working equation: agent = model + harness. The model generates; the harness is everything around it that turns generation into reliable, on-standard output.

The harness makes one promise above all: intent without loss. What the builder means is written down as a contract in phase 1; every later phase is graded against that contract; drift from it is a defect. Speed comes from automation — quality comes from the contract surviving all the way to shipped UI.

  • Normative source: the TFX Design Standard v0.1 (owner: Wondo Jeong). WCAG 2.2 AA is the self-imposed accessibility floor; SGDS and GOV.UK are reference points, not rules.
  • The stack (deliberately boring, AI-legible): Base UI components · Radix Colors · shadcn/ui default tokens · Plus Jakarta Sans (600) display · Inter (400/500/600) body · Teacher & School Blue #0064FF.
  • Brand essence: Kind Utility — useful first, kind at the surface. The one test: does this help teachers work faster with less stress?

Install #

The harness ships as a Claude Code plugin. In a product repo (TW, CaseSync, Glow):

/plugin marketplace add transformteamsg/tfx-design-standard
/plugin install tfx-design-harness@tfx

This installs the five skills, the tfx-design-evaluator subagent, and the control catalog — the catalog ships with the plugin, not with the product repo. Working on the harness itself? Open a Claude Code session in this repository; skills load from .claude/skills/ automatically.

Adopting the harness in a product repo? Follow the step-by-step team onboarding guide (maps the harness-ready checklist to concrete steps).

Architecture #

Three layers, separated so the harness stays reusable across the whole portfolio:

LayerWhat lives thereMechanism
Normative
what must be true		 The control catalog — 38 controls (22 TFX-DS seed + 6 GovTech accessibility ratchet additions + 10 anti-slop controls), tiered L0/L1/L2 standards/catalog.yaml + detail files
Harness
how the agent works		 Five skills (probabilistic — the agent applies them with judgment) orchestrating the design loop, catalog mechanics, onboarding, and voice .claude/skills/
Enforcement
proof, not promises		 Deterministic check scripts (non-skippable), the evaluator subagent for judgment calls, and human gates at plan approval and L1 waivers checks/ + .claude/agents/

The ratchet: every failure caught downstream becomes a new control upstream. The catalog only grows from observed failures — evidence in, rule out, never speculation. Two pilot runs have already fed two controls back into the catalog.

One catalog serves the whole portfolio. Per-product difference is nuance calibration (accent, illustration, tone weighting), never separate rules or systems.

Control catalog #

source of truth: standards/catalog.yaml · format spec: standards/README.md

Every standard is a control: one verifiable statement with an id, a tier, and a check type. The litmus test: if you can't check it, it's a principle or a guideline — not a standard. The index is always loaded; detail files (rationale, pass/fail examples, evaluator guidance) load only when a control is in scope.

- id: TYP-2
  source: TFX-DS
  title: Body text at least 14px; labels at least 11px; body line-height 1.5-1.6
  tier: L1                      # L0 | L1 | L2
  check: deterministic          # deterministic | judgment | hybrid
  phase: [implement, verify]    # intent | plan | implement | verify
  applies_to: [page, component] # page | component | flow | content
  verify: "Type-scale scan; checks/type-scan"
  waiver: documented            # follows tier — see below
  detail: controls/typ-2.md     # omit if the index entry is self-sufficient

The design loop #

Six phases; controls filter into every phase; two moments are human gates. Modifications and catalog re-audits run a scoped version of the same loop (see tfx-design-ui).

PhaseWhat happensGate
1 · IntentPurpose, the teacher and the moment this serves, page type, done-criteria — written as a sprint contract the evaluator later grades against. Naming checked (CNT-2) before a placeholder spreads.—
2 · Diverge2–3 structurally different options, no pixel code, composed only from components that exist.User picks a direction
3 · PlanComponents per region, content outline, async states with their announcement channels, controls in scope, proposed waivers, tradeoffs named. Written to a decision record.HUMAN GATE plan approved before any code
4 · ImplementBuild exactly the approved plan. Manifest components, semantic tokens, the two typefaces, every field labelled. Drift from the plan is itself a defect.—
5 · VerifyDeterministic checks first (L0 blocks, L1 loops back) → screenshots as evidence → separate evaluator grades judgment controls + design quality against the contract.SPLIT generator never grades its own work
6 · RatchetDecision record completed; any defect no control covered becomes a proposed control.Design lead approves new controls

Skills #

Skills are the probabilistic half of the harness — procedures the agent applies with judgment; the checks/ scripts are the deterministic floor underneath. Reliability comes from the combination, never from either alone. Each section below documents one skill and links to its live definition — this page describes them, it does not copy them, so it cannot drift.

tfx-design-ui #

entry point · .claude/skills/tfx-design-ui/SKILL.md

Summary: orchestrates every design change end-to-end. Triggers when anyone asks to design, create, build, add to, change, fix, or restyle a page, screen, form, component, or flow.

Three entry points, same gates

  • New page or flow — all six phases.
  • Modification — scoped loop: one-line intent, skip diverge if structure is fixed, short plan naming only the controls the changed surface pulls in. "Just add a field" is still in scope — a new field still triggers labels, tokens, async states.
  • Catalog update re-audit — when controls are added, shipped surfaces are silently out of date; the "change" is the catalog delta, the plan is the audit findings.

What it enforces beyond the phases

  • Gate protocol — the plan is presented in full first; approval is asked in plain text at the end and needs an explicit typed answer. Unattended runs require up-front operator authorization, recorded honestly as proxy approval.
  • Evidence rules — two screenshot sets: widths (360/768/1280) and one frame per state each hybrid control asserts, loading included. Demo hooks (e.g. ?fail=1) are built during implement so every state is photographable.
  • Honesty about enforcement — controls with no built check are reported "verified manually", never "passed". The manual accessibility pass walks the catalog's A11Y controls in id order.
  • Evaluator dispatch — verify spawns the separate evaluator with the contract, plan, screenshots, and standards path. If the agent can't spawn subagents, its orchestrator dispatches instead. The verdict lands in the decision record verbatim — a summary is a defect.

tfx-design-standards #

rulebook · .claude/skills/tfx-design-standards/SKILL.md

Summary: catalog mechanics — how to read, filter, apply, and grow the catalog. Triggers when applying controls, judging a deviation, writing a waiver, or proposing a control; loaded automatically by tfx-design-ui.

Reading & filtering

  • The catalog resolves relative to the skill itself — it ships with the plugin, not the product repo. Filter by phase and applies_to: a copy edit pulls content controls, not all 28.
  • A control without a detail: file is self-sufficient; a judgment/hybrid control missing one is a catalog defect to raise, not a rubric to improvise.

Waiver protocol

  • The agent may propose L1 waivers; only a named human grants them. L2 deviations carry an inline tfx-waive with a specific reason. Content that can't host a comment (i18n, CMS) records the waiver in the decision record instead.
  • When a control seems wrong: check its "Do not flag" exceptions → propose a waiver at the right gate → surface the conflict. Never edit the catalog to make a failing check pass.

Growing the catalog

  • Evidence in, rule out. Lightweight PR + design-lead approval (workflow in CONTRIBUTING.md); high bar for L0/L1, evidence bar for L2.
  • Ratified additions must name their re-audit set — which shipped surfaces the new control affects.

tfx-content-style #

copy · .claude/skills/tfx-content-style/SKILL.md

Summary: TFX voice & tone applied at generation time, so copy ships on-voice and nobody — least of all non-native English speakers — carries the UX-writing burden personally. Sufficient alone for copy-only edits; tfx-design-ui pulls it in at implement.

Voice & tone

  • Voice (constant): warm not sappy · clear not cold · helpful not pushy · professional not stiff · confident not arrogant.
  • Tone (by context): success — celebratory, brief · error — calm, helpful · onboarding — encouraging · destructive — sober, no drama · empty state — inviting, lead with the next action.

The binding rules

  • Mechanics (CNT-3) — second person, active voice, sentences ≤ 25 words; Singapore English spelling; read it aloud — if it sounds robotic, rewrite it.
  • Errors (CNT-1) — what happened → what it means → what to do next; say what happened to their data ("Your marks are saved on this device"); never a raw code as the primary message; never blame the user.
  • Naming (CNT-2) — name by function in words teachers already use: "Class Planner" yes, "SyncFlow" no. A name that needs explaining has already failed.

Calibration & conflicts

  • Same character everywhere, calibrated weight: TW calm and steady · CaseSync reserved, privacy-forward · Glow warmer, more celebratory.
  • Statutory or ministry-mandated wording wins — waive it, don't reword it.

tfx-design-review #

rubric · .claude/skills/tfx-design-review/SKILL.md

Summary: the grading procedure for finished design work, used by the tfx-design-evaluator subagent — not by the agent that produced the design. The grader returns findings with evidence, never patches.

Four grading passes

  1. Contract compliance — each done-criterion: met / not / partially, with quoted evidence.
  2. Plan fidelity — the build matches the human-approved plan; drift is a finding even when the drifted version looks fine.
  3. Judgment controls — each in-scope control graded via its detail file's "Evaluator guidance" and "Do not flag" sections.
  4. Quality criteria — design quality · originality (inverted for professional tools: unwarranted novelty is flagged as readily as generic slop) · craft · functionality.

Output contract

VERDICT: pass | pass-with-findings | fail
BLOCKING:                # must fix before ship
ADVISORY:                # should fix
QUALITY GRADES:          # four criteria, with reasons
JUDGMENT CONTROL NOTES:  # one line per control, evidence quoted;
                         # CMP-1 always names its evidence source
UNCOVERED:               # defects no control covers — feeds the ratchet

Calibration: a screening pass, not final authority — decisive on clear violations; close calls escalate to a human instead of manufacturing confidence.

tfx-design-evaluator #

subagent · .claude/agents/tfx-design-evaluator.md

Summary: the agent definition that runs tfx-design-review — spawned at verify with the contract, approved plan, screenshots, in-scope controls, and the absolute standards path.

Why it's built this way

  • Preloaded rubric (skills: tfx-design-review) — the procedure is injected at spawn, not left for the agent to maybe read.
  • Pinned model (model: opus) — grading quality doesn't silently follow whatever the main session runs.
  • Scoped tools (Read, Grep, Glob, Bash) — it can inspect everything and edit nothing.
  • Honest limits, stated in its own prompt — same model and standards as the generator, so it is a rigorous second read, not full independence; what it can't verify from the evidence, it says so and routes to a human.

Why a separate evaluator at all? Self-evaluation skews positive — an agent grading its own design finds what it hoped to build, not what it built. Track record across three pilot reviews: a copy-vs-behavior gap, a non-firing 360px breakpoint, stranded keyboard focus, and a double screen-reader announcement — two of its findings became catalog controls.

Checks & commands #

scripts & status: checks/README.md

Every check exits 0 on pass and 1 on violation, printing file:line, the control id, and a fix suggestion — verbose on failure, silent on success. Inline waivers never silence L1 controls; they downgrade the message to [waiver-claimed] and still fail until a human verifies the approver.

CommandStatusCovers
python3 checks/validate.py BUILT The harness's own verification baseline: catalog schema, tier→waiver pairing, detail-file ↔ catalog consistency, and every control id referenced anywhere. Run before committing any standards/ change.
python3 checks/token-audit.py <path> BUILT TOK-1..3 + COL-1..2: raw colours, off-scale spacing/radius, Tailwind palette bypasses. Token-definition blocks are exempt. 18-case self-test (--self-test).
contrast focus labels targets reduced-motion alt-scan structure nrv title-lang skip-link announce type-scan … PLANNED The remaining deterministic floor (V1) — wired as hooks during implement and as the verify gate once calibrated against a real product repo.

Decision records #

One record per page or significant change — started at the plan gate (the approved plan is the fixed artifact verify grades against), finished at the ratchet. It holds the sprint contract, tradeoffs, waivers with approvers, the verbatim evaluator verdict, and any control proposals.

  • Template — copy per page/change.
  • Worked examples from the pilot runs: Student Notes empty state · Attendance (includes a catalog re-audit and two evaluator re-reviews).
  • Implemented pilot pages with full state evidence: student-notes · attendance (?fail=1 demos the error state) · friction report from run 1.
