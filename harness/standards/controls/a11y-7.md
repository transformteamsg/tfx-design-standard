## Requirement

Express the page's structure in markup, not just visually: real heading elements in
a logical hierarchy, lists as `ul`/`ol`, tabular data as tables, related form fields
grouped. Headings and labels describe the content they introduce; link text makes
sense read on its own.

## Rationale

GovTech checklist Essential tier ("Presentation of info and relationships" +
"Headings and labels" + "Link text and purpose"). A teacher using a screen reader
navigates by headings and landmarks — visual-only structure is invisible to them.
This is also what makes pages legible to agents and checks: semantic markup is
machine-readable structure.

## Passes when

- One `h1` per view; heading levels nest without skipping; sections that look like
  sections are sectioned.
- A roster rendered as a list IS a list (`ol`/`ul`); data grids are tables with
  header cells.
- Headings name their content ("Attendance", "No notes yet"), not generic
  ("Overview", "Section 2").
- Link text describes the destination ("Make a correction"), never "click here".

## Fails when

- Styled `div`/`span` standing in for headings, lists, or buttons.
- Heading levels chosen for font size rather than hierarchy.
- A label that only makes sense with visual context ("More", "Here", bare icons).

## How to verify

Deterministic half — planned `checks/structure`: heading-hierarchy walk, list/table
semantics scan. Until built: verify manually against this file. Judgment half — the
evaluator reads headings/labels/links out of visual context and judges whether they
still communicate.
