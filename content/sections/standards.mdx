Principles tell you how to decide; guidelines tell you what usually works. Standards are different: they are **requirements**. If a screen violates one, it doesn't ship — no matter who or what built it.

Each standard is a **control**: one verifiable statement with explicit fail conditions. "Text meets WCAG AA contrast" is a control; "interfaces should feel calm" is not — that's a principle, because no machine can check it. This checkability is the whole trick: it means the bar can be enforced before review, automatically.

The same catalog binds both audiences:

- **Humans** read it rendered in the [control catalog](/standards/catalog), and cite control IDs in reviews ("this fails SLP-3").
- **AI agents** read the identical catalog as [YAML](/standards/catalog.yaml) and apply it while generating UI.

Controls come in three tiers:

- **L0 — non-negotiable.** Accessibility floors, destructive-action safety. No waiver exists; a violation blocks ship.
- **L1 — mandatory.** Binding by default. Deviation requires a documented waiver in code: `tfx-waive <ID> reason="<specific reason>"`.
- **L2 — recommended.** Follow unless you have a better answer; deviating needs an inline rationale.
