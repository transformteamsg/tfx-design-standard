Everything on this site exists in three forms, for three readers.

**Humans read the site.** Principles, guidelines, and foundations are written for people making judgement calls. Browse, decide, deviate with a reason.

**Humans hand skills to machines.** Skills in the [harness](/harness) are markdown instructions: human-readable guidelines packaged so an agent can execute them. You read a skill to trust it; the agent reads it to apply it. Skills don't define the bar, they carry it.

**Machines read the contract.** The control catalog is the one layer precise enough to enforce automatically, so it is published twice: rendered for review at [/standards/catalog](/standards/catalog), and as data at [/standards/catalog.yaml](/standards/catalog.yaml).

Every page on this site is also available as Markdown: append `.md` to its path. So [/guidelines/voice-tone](/guidelines/voice-tone) has a clean Markdown twin at `/guidelines/voice-tone.md` — the page content, no site chrome. Fetch one page when you need one page.

## Wiring it into your tool

Three machine surfaces, depending on how much you need:

- [/llms.txt](/llms.txt) is the curated index. It lists every page and links each `.md` twin. Start here to see what exists.
- [/llms-full.txt](/llms-full.txt) is every page concatenated, plus per-control rationale and examples. Paste it when you want the whole standard in context.
- [/standards/catalog.yaml](/standards/catalog.yaml) is the control source. Point your tool at it when you only need the controls.

Tiers are binding: L0 cannot be waived, L1 needs `tfx-waive <ID> reason="..."` in code, L2 deviations need an inline rationale.

One thing the catalog is not: a skill. Skills apply the standard inside a session; the catalog *is* the standard. When they disagree, the catalog wins, and the skill gets fixed.
